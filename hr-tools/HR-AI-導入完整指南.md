## 1. 落地型 AI 基礎知識

### 1.1 什麼是「落地型 AI」？

**定義**：能夠真正在企業內部執行實際工作、產生可量化業務價值的 AI 系統。

**與「聊天機器人」的差異**：

| 特性 | 傳統聊天機器人 | 落地型 AI |
|------|---------------|-----------|
| 功能 | 回答問題 | **執行任務** |
| 數據 | 通用知識庫 | **企業私有數據** |
| 整合 | 獨立系統 | **串接既有系統**（Email, CRM, Sheets） |
| 產出 | 文字回覆 | **可交付成果**（履歷池、報表、自動信件） |
| 學習 | 預訓練模型 | **持續學習企業流程** |

**HR 場景的「落地」標準**：
- ✅ 能自動處理履歷進件（非手動轉發）
- ✅ 能主動搜尋候選人（非被動等待）
- ✅ 能生成可用的招募報告（非僅提供建議）
- ✅ 能追蹤 Pipeline 並提醒跟進（非僅記錄）

---

### 1.2 AI 導入成熟度模型

**Level 0：人工作業**
- 履歷用 Email 收，手動整理到 Excel
- 候選人手動搜尋、手動聯繫
- 報告用 Word 手寫

**Level 1：AI 輔助**（大多數企業現況）
- 用 ChatGPT 寫 JD、改履歷
- AI 提供招募建議，HR 執行
- 數據仍需人工整理

**Level 2：流程自動化**（Step1ne 現階段）
- AI 自動解析履歷 → 匯入系統
- AI 主動搜尋候選人 → 配對職缺
- AI 生成週報 → 自動發送

**Level 3：智能決策**（目標）
- AI 預測離職風險 → 提前挽留
- AI 優化薪資結構 → 降低流失率
- AI 識別高潛人才 → 制定培養計畫

**Level 4：自主營運**（未來願景）
- AI 端到端管理招募流程
- AI 自動調整策略（根據市場變化）
- 人類僅做最終決策與異常處理

---

### 1.3 HR 場景 AI 應用分類

**按功能劃分**：
1. **招募（Recruitment）**：履歷解析、候選人搜尋、面試排程
2. **培訓（Learning & Development）**：個人化學習路徑、技能評估
3. **績效（Performance）**：目標追蹤、評估自動化、回饋分析
4. **薪酬（Compensation）**：市場薪資分析、調薪建議
5. **分析（Analytics）**：離職預測、人力配置優化

**按技術類型劃分**：
1. **NLP 自然語言處理**：履歷解析、JD 撰寫、員工回饋分析
2. **機器學習**：候選人配對、離職預測、薪資建議
3. **RPA 流程自動化**：資料搬運、報表生成、Email 發送
4. **對話式 AI**：員工問答、政策諮詢、Onboarding 引導

## 2. 資安與合規

### 2.1 數據隱私法規

**台灣個資法（PDPA）**：
- **第 6 條**：特定目的搜集，不得逾越必要範圍
- **第 8 條**：搜集前需告知當事人（履歷來源、用途）
- **第 19 條**：非公務機關利用個資需取得同意
- **罰則**：違法最高罰 2,000 萬元

**歐盟 GDPR（若涉及歐盟員工/候選人）**：
- **Art. 6**：合法處理基礎（同意、合約、合法利益）
- **Art. 9**：特殊類別資料（健康、種族、性傾向）禁止處理
- **Art. 17**：刪除權（被遺忘權）
- **Art. 22**：自動化決策權（AI 評估需人工審核）
- **罰則**：最高 2,000 萬歐元或全球營收 4%

**關鍵合規要求**：
1. ✅ **告知義務**：候選人需知道其履歷如何被處理
2. ✅ **同意機制**：履歷池存儲需取得明示同意
3. ✅ **數據最小化**：只搜集必要資訊（姓名、經歷、技能）
4. ✅ **存儲期限**：履歷保留期限明確（建議 2 年）
5. ✅ **刪除機制**：候選人可要求刪除資料

---

### 2.2 員工數據保護

**敏感資料分類**：

| 資料類型 | 敏感等級 | 處理原則 |
|---------|---------|---------|
| 姓名、聯絡方式 | 低 | 加密傳輸 |
| 履歷、工作經歷 | 中 | 存取控制 |
| 薪資、績效 | **高** | 強加密 + 審計日誌 |
| 健康、病歷 | **極高** | 禁止 AI 處理（需人工） |

**技術保護措施**：
1. **加密**：
   - 傳輸：TLS 1.3
   - 存儲：AES-256
   - API Key：環境變數，不寫死在程式碼

2. **存取控制**：
   - 角色權限（RBAC）：HR 主管 > 顧問 > AI Agent
   - 最小權限原則：AI 只能讀必要欄位
   - 審計日誌：所有存取留記錄

3. **匿名化**：
   - 統計分析用假名（Pseudonymization）
   - AI 訓練不使用真實姓名

---

### 2.3 AI 決策透明度與公平性

**可解釋性（Explainability）**：
- ❌ 錯誤：「AI 推薦這位候選人」（黑箱）
- ✅ 正確：「AI 推薦這位候選人，因為：技能匹配 90%、經驗 5 年、地點符合」

**公平性（Fairness）**：
- **避免歧視**：年齡、性別、種族不應成為篩選條件
- **檢查偏見**：定期審查 AI 推薦結果是否偏向特定群體
- **人工審核**：關鍵決策（Offer、解僱）不能完全自動化

**透明度要求**：
- 候選人有權知道「AI 如何評估我」
- 員工有權要求「人工重新審查 AI 決策」

---

### 2.4 系統安全架構

**三層防護**：

```
Layer 1: 邊界防護
├─ Firewall（只開放必要 Port）
├─ VPN（遠端存取）
└─ DDoS 防護

Layer 2: 應用防護
├─ OAuth 2.0 認證
├─ API Rate Limiting
├─ Input Validation（防 SQL Injection）
└─ HTTPS Only

Layer 3: 數據防護
├─ 加密存儲
├─ 定期備份（異地）
├─ 審計日誌（不可竄改）
└─ 最小權限
```

**定期檢查**：
- [ ] 每季安全掃描（Vulnerability Scan）
- [ ] 每半年滲透測試（Penetration Test）
- [ ] 每年合規稽核（Compliance Audit）

### 2.5 第三方 API 風險管理

**常用第三方服務風險評估**：

| 服務 | 用途 | 風險等級 | 控制措施 |
|------|------|---------|---------|
| Google Sheets | 履歷池存儲 | 中 | 權限控制、定期備份 |
| Gmail API | 履歷進件 | 中 | OAuth、最小權限 |
| OpenAI API | 履歷解析 | **高** | **數據脫敏**、不傳敏感資料 |
| LinkedIn API | 候選人搜尋 | 低 | 合規使用、避免爬蟲 |

**關鍵原則**：
1. **數據脫敏**：傳給 OpenAI 前移除姓名、電話、身分證字號
2. **合約審查**：確認第三方 SLA、數據所有權、刪除機制
3. **備援計畫**：主要 API 掛掉時的替代方案

## 3. HR AI 導入方法論

### 3.1 導入前評估

**評估框架（5 個維度）**：

**1. 業務需求**
- [ ] 現有痛點明確？（手動處理履歷耗時？候選人來源不足？）
- [ ] 有量化目標？（減少 50% 處理時間？提升 30% 配對準確度？）
- [ ] 預算與時程？（3 個月內上線？預算 50 萬？）

**2. 數據就緒度**
- [ ] 現有數據格式？（Excel？CRM？紙本？）
- [ ] 數據品質？（完整性、一致性、準確性）
- [ ] 數據量？（至少 100 筆履歷才能訓練 AI）

**3. 技術可行性**
- [ ] 現有系統能整合？（有 API？能匯出數據？）
- [ ] 技術團隊能力？（有工程師？能維護？）
- [ ] 基礎設施？（雲端？地端？網路頻寬？）

**4. 組織準備度**
- [ ] 高層支持？（預算核准？願意調整流程？）
- [ ] 團隊接受度？（HR 願意學習新工具？）
- [ ] 變革管理？（有專人負責推動？）

**5. 合規風險**
- [ ] 法規檢查完成？（個資法、勞基法、GDPR？）
- [ ] 數據保護措施？（加密、權限、審計？）
- [ ] 員工告知機制？（候選人知情同意？）

---

### 3.2 技術選型框架

**選型決策樹**：

```
Q1: 預算？
├─ < 10萬 → 開源方案（OpenClaw + 免費 API）
├─ 10-50萬 → 混合方案（開源框架 + 付費 API）
└─ > 50萬 → 商業方案（客製化開發 + 企業級支援）

Q2: 技術能力？
├─ 無工程師 → SaaS 方案（BambooHR, Workday）
├─ 有 1-2 人 → Low-Code 平台（Zapier + Airtable）
└─ 有團隊 → 完全客製化（OpenClaw + 自建）

Q3: 數據敏感度？
├─ 高（薪資、績效）→ **地端部署**
├─ 中（履歷）→ 混合雲（核心地端、處理雲端）
└─ 低（公開職缺）→ 全雲端

Q4: 擴展性需求？
├─ 單一部門（<50人）→ 簡單工具
├─ 全公司（50-500人）→ 模組化系統
└─ 集團（>500人）→ 企業級平台
```

## 4. 完整選育用留落地方案

### 4.1 選（招募）- 完整流程

#### 階段 1：需求分析

**JD（職缺描述）管理**：
- **輸入**：部門主管需求、人力預算
- **AI 處理**：
  1. 分析歷史 JD → 提取常用技能關鍵字
  2. 建議職位名稱、薪資區間（參考市場數據）
  3. 生成結構化 JD（職責、要求、福利）
- **輸出**：標準化 JD + Google Sheets 職缺清單

**工具實現**（Step1ne 經驗）：
```bash
# JD 管理腳本
~/clawd/hr-tools/jd-manager.sh add "AI工程師" "技術部" 2 "80k-120k" "Python,TensorFlow" "3年+" "資工碩士"
```

**職缺追蹤**：
- 狀態：草稿 → 招募中 → 已填補 → 已關閉
- 欄位：職位、部門、人數、薪資、技能、經驗、學歷、狀態、建立日期

---

#### 階段 2：候選人搜尋

**多管道自動化**：

**管道 1：LinkedIn 公開搜尋**
- **關鍵字組合**：職位 + 技能 + 地點
- **篩選條件**：經驗年數、公司規模、學歷
- **產出**：姓名、公司、職位、LinkedIn URL

**管道 2：GitHub 開發者搜尋**（技術職缺）
- **搜尋條件**：語言（Python, Java）、專案 Star 數、活躍度
- **評估指標**：Commit 頻率、PR 品質、技術棧
- **產出**：GitHub ID、技能標籤、專案連結

**管道 3：履歷池內部搜尋**
- **全文檢索**：技能、經驗、公司、學校
- **優先級**：P0（完美匹配）> P1（可培養）> P2（備選）
- **產出**：候選人清單 + 配對分數

**管道 4：社群挖掘**（進階）
- **目標平台**：Kaggle（數據科學）、Medium（技術寫作）、GitHub Discussions
- **識別指標**：影響力、專業度、活躍度

**自動化腳本**（Step1ne 實戰）：
```bash
# 每週一 10:00 自動搜尋
~/clawd/hr-tools/auto-sourcing-final.sh
```

#### 階段 3：履歷解析與配對

**履歷解析流程**：

1. **文件處理**：
   - 支援格式：PDF, DOCX, TXT, HTML
   - OCR 處理（掃描件）：Tesseract
   - 轉純文字：pdf2txt

2. **資訊提取**（NLP）：
   ```
   姓名：王小明
   電話：0912-345-678
   Email：ming@example.com
   學歷：台大資工碩士
   經驗：
     - Google 軟體工程師 (2020-2023)
     - 微軟實習生 (2019-2020)
   技能：Python, TensorFlow, Kubernetes
   ```

3. **結構化存儲**（Google Sheets）：
   - 欄位：姓名、聯絡方式、學歷、經驗、技能、當前職位、狀態、來源、日期

**AI 配對評分**：

```python
def match_score(candidate, job):
    score = 0
    # 技能匹配（40%）
    skill_match = len(set(candidate.skills) & set(job.required_skills)) / len(job.required_skills)
    score += skill_match * 40
    
    # 經驗年數（30%）
    exp_match = 1 if candidate.years >= job.min_years else candidate.years / job.min_years
    score += exp_match * 30
    
    # 學歷（20%）
    edu_match = 1 if candidate.degree >= job.min_degree else 0.5
    score += edu_match * 20
    
    # 地點（10%）
    loc_match = 1 if candidate.location == job.location else 0
    score += loc_match * 10
    
    return score
```

**產出範例**：
```
AI工程師 職缺配對結果（2026-02-15）
找到 30 位候選人：
- P0（90分以上）：3人
  1. 王小明（95分）- Google 3年經驗、台大碩士、Python/TF 專家
  2. 李小華（92分）- Meta 2年經驗、清華碩士、熟 MLOps
  3. ...
- P1（70-89分）：12人
- P2（50-69分）：15人

已匯入履歷池：行 150-180
```

**Offer 生成**：

### 5.4 與既有系統整合

**整合策略**：

**方案 1：API 整合**（優先）
- **適用**：系統有 REST API（如 BambooHR, Workday）
- **方式**：直接呼叫 API、雙向同步
- **優點**：即時、可靠、官方支援
- **範例**：Google Sheets API, Gmail API

**方案 2：資料匯入/匯出**（次選）
- **適用**：系統支援 CSV/Excel 匯出
- **方式**：定期匯出 → AI 處理 → 匯入回系統
- **優點**：不需改動既有系統
- **範例**：每週匯出員工清單 → AI 分析 → 產生報告

**方案 3：資料庫直連**（謹慎）
- **適用**：有資料庫存取權限
- **方式**：直接讀寫資料庫
- **風險**：可能破壞數據一致性、需專業 DBA 協助
- **範例**：讀取離職率數據（唯讀）

**方案 4：RPA 自動化**（最後手段）
- **適用**：系統無 API、無匯出功能
- **方式**：模擬人工操作（點擊、輸入）
- **缺點**：脆弱、維護成本高
- **範例**：agent-browser 自動填寫舊系統表單

### 6.2 常見問題與解決方案

**問題 1：AI 解析履歷錯誤**
- **現象**：姓名、電話提取錯誤，或漏掉重要資訊
- **原因**：履歷格式多樣（PDF 掃描件、特殊排版）
- **解決**：
  1. 用 GPT-4（更強理解力）替代 GPT-3.5
  2. Prompt 優化：提供範例格式
  3. 人工複查機制：錯誤率 >10% 時通知

**問題 2：Google Sheets 換行符號錯亂**
- **現象**：1 筆資料變多行，欄位錯位
- **原因**：數據中的 `\n` 被當成「新一行」
- **解決**：
  ```python
  # 匯入前清理
  data = data.replace('\n', ' ').replace('\r', ' ')
  data = data.replace(', ', ' | ')  # 逗號也會被當換行
  ```

**問題 3：LinkedIn/104 反爬蟲**
- **現象**：請求被封鎖、IP 被 ban
- **原因**：頻率過高、未模擬人類行為
- **解決**：
  1. Layer 1：隨機延遲 2-5 秒/操作
  2. Layer 2：批量間隔 120 秒（<30 次/小時）
  3. Layer 3：使用官方 API（付費但穩定）

**問題 4：數據重複**
- **現象**：同一候選人匯入多次
- **原因**：未做去重檢查
- **解決**：
  ```python
  # 匯入前檢查
  existing = sheets.get_column('姓名')
  if candidate.name in existing:
      skip()
  ```

**問題 5：成本失控**
- **現象**：OpenAI API 帳單爆增
- **原因**：大量呼叫、未優化 Prompt
- **解決**：
  1. 快取機制：相同問題不重複問 AI
  2. 模型降級：簡單任務用 GPT-3.5
  3. Prompt 精簡：去掉冗餘描述
