## 共用開場問題（每次訪談必問）

> 在進入職位細節前，先建立對公司與用人主管的整體理解。

**Q0-1**：請問這次和我們合作的主要聯絡人是誰？他/她在公司的職銜與決策權限是什麼？
- **目的**：確認 decision maker，避免最後被告知「還要再讓另一個人決定」。

**Q0-2**：仁大資訊目前在哪些客戶產業最活躍？這個新職位主要服務哪類客戶專案？
- **目的**：了解業務重心，幫助定位候選人的產業背景需求。

**Q0-3**：這 9 個職位中，哪幾個是最急迫的？優先順序如何？
- **目的**：資源分配，獵頭要先集中火力在最緊急的職缺。

## 一、PM（專案經理）× 2人

**組織背景**

**Q1**：PM 目前團隊有幾位？這兩位新 PM 進來後，會各自負責不同客戶專案，還是協作同一個大專案？
- **目的**：釐清工作結構，判斷候選人需要「獨立作業」還是「協作」能力。

**Q2**：PM 直接匯報給誰？是技術主管還是業務/客戶成功部門主管？
- **目的**：了解 PM 的定位是偏技術端還是客戶端，影響候選人篩選方向。

**職位核心職責**

**Q3**：一位 PM 平均同時管理幾個專案？每個專案的規模大概多少（人月、預算）？
- **目的**：評估候選人過去的案子規模是否匹配，避免 over/under-qualified。

**Q4**：KPI 是怎麼設定的？是以「準時交付率」為主，還是「客戶滿意度」或「成本控制」？
- **目的**：幫助候選人理解成功標準，也讓獵頭找到真正能達標的人。

**必備 vs 加分條件**

**Q5**：C#、JavaScript、SQL 這些技術要求，是希望 PM 能「看懂」還是「自己寫」？
- **目的**：技術 PM vs 管理 PM 有很大差距，需明確。

**Q6**：PMP 認證是加分，但如果候選人沒有 PMP 但有 5 年以上實戰經驗，你們會考慮嗎？
- **目的**：了解彈性空間，避免好人選因證照被刷掉。

**軟性需求**

**Q7**：你們的客戶以哪類產業為主（金融、製造、政府）？PM 需要具備哪些行業知識？
- **目的**：某些客戶有特殊溝通文化（例如政府機關），需要有類似經驗的人。

**Q8**：過去在 PM 身上最常看到的「失敗特質」是什麼？最想避開哪類人？
- **目的**：反向問法，精準理解用人主管的雷區，減少推薦失誤。

**薪資框架**

**Q9**：「待遇面議」的實際範圍是多少？優秀的人有機會拿到多少？有年終獎金嗎？
- **目的**：務必確認真實薪資帶，避免後期 offer 談崩。

**招募時程**

**Q10**：預計面試幾關？決策者是用人主管本身，還是需要 HR 或更高層簽核？
- **目的**：掌握時間軸，才能給候選人合理的期望值管理。

**過去招募經驗**

**Q11**：之前招 PM 有遇到什麼挑戰？是找不到人、還是找到了但留不住？
- **目的**：了解真正的痛點，調整搜尋策略（例如若留不住，要問文化適配性）。

**補充確認**

**Q12**：出差頻率如何？主要出差地點在哪裡（台灣各縣市 or 海外）？
- **目的**：出差是很多候選人的 deal-breaker，要提前確認。

**Q13**：試用期多長？轉正標準是什麼？
- **目的**：候選人常問，先備好答案顯得專業。

**Q14**：這個職位預計什麼時候要到職？
- **目的**：對齊時程，避免好候選人等太久而接了其他 offer。

**Q15**：如果我們推薦的候選人非常優秀但背景稍微偏，你們有調整 JD 的彈性嗎？
- **目的**：探測彈性空間，為後續推薦爭取主動權。

## 二、SD（系統設計師）× 1人

**組織背景**

**Q1**：SD 在組織架構中位於哪個層級？是獨立存在還是隸屬於開發團隊之下？
- **目的**：了解 SD 是「橋接角色」還是「技術權威角色」。

**Q2**：SD 主要配合的對象是 SA、PG，還是直接與客戶互動？
- **目的**：決定候選人需要的溝通能力層次。

**職位核心職責**

**Q3**：一個典型的專案週期中，SD 介入的時間點是什麼？從需求確認後，還是更早？
- **目的**：理解工作節奏，找到有類似工作模式的候選人。

**Q4**：SDD（系統設計文件）的產出頻率如何？文件標準有固定格式嗎？
- **目的**：文件能力是 SD 核心，確認候選人文件經驗是否匹配。

**必備 vs 加分條件**

**Q5**：微服務、Docker、Kubernetes 是「最好有」還是「沒有就不考慮」？
- **目的**：區分 must-have 和 nice-to-have，擴大或縮小人才池。

**Q6**：SQL 和資料表設計需要到什麼程度？需要能自己 optimize query 嗎？
- **目的**：確認技術深度要求，避免推薦的人技術底子不夠。

**軟性需求**

**Q7**：你們期待 SD 在 Code Review 中扮演什麼角色？是標準把關者，還是教練/導師？
- **目的**：不同的角色期待，對候選人的溝通風格有不同要求。

**Q8**：SD 需要主動提出設計方案，還是主要是接受 SA 的需求後執行設計？
- **目的**：主動型 vs 執行型人才需求差異很大。

**薪資框架**

**Q9**：55K–80K 這個區間，通常幾年經驗的人會落在哪個區間？
- **目的**：讓候選人對薪資期望有更精準的理解。

**招募時程**

**Q10**：面試流程會有技術測試嗎（筆試/白板題/實作題）？
- **目的**：讓候選人提前準備，降低因緊張導致的失誤。

**過去招募經驗**

**Q11**：上一位 SD 為什麼離開？這個職位的挑戰性在哪裡？
- **目的**：了解職位真實難度，也可判斷是否有留才問題。

**補充**

**Q12**：SD 有機會接觸到雲端架構（AWS/Azure）嗎？還是主要是 on-premise 環境？
- **目的**：雲端經驗是市場敏感指標，影響候選人的市場價值判斷。

**Q13**：這個角色有帶人的需求嗎？現在或未來？
- **目的**：有些 SD 候選人有強烈帶人意願，有些完全不想，要提前對齊。

**Q14**：期望到職日？
- **目的**：基本時程確認。

**Q15**：你們怎麼定義「一位優秀的 SD」？給我描述一個你心目中的理想人選。
- **目的**：開放式問題，讓主管說出真正的期待，往往能挖出 JD 沒寫的關鍵需求。

## 三、SRE（系統維運工程師）× 2人

**組織背景**

**Q1**：SRE 團隊目前規模如何？這兩位新人進來後，負責的系統範圍會如何分配？
- **目的**：了解工作範疇，判斷候選人需要「廣度」還是「深度」。

**Q2**：SRE 主要維運的是客戶的系統，還是內部平台？On-call 制度是怎麼安排的？
- **目的**：On-call 對許多工程師是 deal-breaker，要提前說清楚。

**職位核心職責**

**Q3**：CI/CD pipeline 目前用什麼工具？SRE 需要從頭建置，還是維護既有的？
- **目的**：「建置」和「維護」對候選人的技術要求差異很大。

**Q4**：Kubernetes 環境是自建（on-premise）還是 managed（EKS/AKS/GKE）？
- **目的**：不同環境需要不同的技術棧，影響人才篩選。

**必備 vs 加分條件**

**Q5**：Linux 和 Windows Server 兩個都要熟，還是其中一個為主？
- **目的**：澄清主要作業環境，避免只熟一邊的好人選被刷掉。

**Q6**：Python/Bash/PowerShell 腳本能力，需要到能獨立開發自動化工具的程度嗎？
- **目的**：腳本能力差距很大，要明確 level。

**軟性需求**

**Q7**：這個職位面對的最大挑戰是什麼？是系統複雜度、客戶壓力，還是技術債？
- **目的**：讓候選人提前心理準備，也幫顧問找到心理素質匹配的人。

**Q8**：你們更看重「穩定性/可靠性導向」還是「快速迭代/創新導向」的工程師？
- **目的**：SRE 文化分兩種，候選人的工作風格要吻合。

**薪資框架**

**Q9**：65K–95K 的區間，具體取決於哪些因素？年資？技術深度？特定認證？
- **目的**：讓顧問能精準告知候選人薪資期望。

**招募時程**

**Q10**：這個職位的面試會有實際的技術情境模擬（如：給一個系統架構問題現場分析）嗎？
- **目的**：讓候選人知道面試型態，也讓顧問篩選出能應對技術面試的人。

**過去招募經驗**

**Q11**：SRE 人才在市場上不好找，你們之前用過哪些管道？遇到最大的困難是什麼？
- **目的**：了解市場現實，也讓客戶理解獵頭的價值所在。

**補充**

**Q12**：出差的目的地通常是哪裡？頻率如何？
- **目的**：SRE 可能需要到客戶機房，了解出差性質。

**Q13**：這個職位有資安相關的工作內容嗎（如：漏洞掃描、合規）？
- **目的**：資安跨界能力是加分，若有需求要提前說。

**Q14**：期望到職日？試用期轉正標準？
- **目的**：基本時程和考核確認。

**Q15**：如果推薦的候選人背景偏 DevOps 而非純 SRE，你們有彈性嗎？
- **目的**：DevOps/SRE 界線模糊，確認客戶的彈性有助於擴大人才池。
