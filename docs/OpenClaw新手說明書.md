## 1. 什麼是 OpenClaw？

### 簡單說

OpenClaw 是一個讓 AI（Claude、GPT、Gemini 等）變成「會做事的助理」的系統。

### 和 ChatGPT 有什麼不同？

| 功能 | ChatGPT | OpenClaw |
|------|---------|----------|
| 對話 | ✅ | ✅ |
| 記憶 | ❌ 每次重開都忘記 | ✅ 有長期記憶 |
| 執行任務 | ❌ 只能聊天 | ✅ 能執行程式、寫檔案、發信 |
| 主動性 | ❌ 等你問 | ✅ 能定時自動檢查、主動報告 |
| 整合 | ❌ 孤立 | ✅ Gmail、Sheets、Telegram 都能連 |

**核心差異**：ChatGPT 是「對話工具」，OpenClaw 是「AI 員工」。

```bash

openclaw --version
```

### Step 2: 啟動 Gateway

```bash
openclaw gateway start
```

### Step 3: 建立工作區

```bash
mkdir ~/my-ai-assistant
cd ~/my-ai-assistant
```

### Step 4: 建立核心檔案

OpenClaw 需要 4 個核心檔案：

#### `USER.md` - 關於你

```markdown
# USER.md

- **Name:** [你的名字]
- **Timezone:** Asia/Taipei
- **Telegram:** @[你的帳號]

## 合作方式
- 希望 AI 當「AI 員工」，主動推進專案
- 里程碑式交付：做完一段 → 確認 → 下一段
- 回報簡潔：預設 3 行，除非我說「解釋」
```

#### `SOUL.md` - AI 的人設

```markdown
# SOUL.md

## 我是誰
- 我是你的 AI 助理
- 我像「員工」而不是「秘書」
- 我主動解決問題，不只等指令

## 我的原則
1. 結果導向 — 只看產出，不說過程
2. 里程碑交付 — 做完一段再報告
3. 3 行原則 — 預設簡潔
4. 主動記錄 — 所有決策寫在檔案
```

#### `AGENTS.md` - 工作規則

```markdown
# AGENTS.md

## 每次啟動時

1. 讀取 USER.md（了解你是誰）
2. 讀取 SOUL.md（了解我是誰）
3. 讀取 memory/今天.md（了解今天要做什麼）
4. 讀取 MEMORY.md（了解長期記憶）

## 記憶系統

- **每日日誌**：`memory/YYYY-MM-DD.md`
- **長期記憶**：`MEMORY.md`

所有重要決策、學習、進度都要記錄。
```

#### `MEMORY.md` - 長期記憶

```markdown
# MEMORY.md

## 核心規則

- 里程碑交付制
- 每天 21:00 自動日報
- 你說「OK」= 授權繼續

## 當前專案

（專案清單）

## 重要決策記錄

（決策記錄）
```

## 4. 核心概念（必讀）

### 概念 1：AI 每次重啟都會「失憶」

**問題**：
AI 沒有「記憶」，每次對話都是全新的。

**解決方案**：
所有重要資訊寫在檔案裡。

#### 記憶系統

```
memory/
├── 2026-02-10.md   ← 今天的日誌
├── 2026-02-09.md   ← 昨天的日誌
└── ...

MEMORY.md            ← 長期記憶（重要決策、學習）
```

**規則**：
- 每天的工作記錄在 `memory/YYYY-MM-DD.md`
- 重要的東西（決策、模式、學習）記錄在 `MEMORY.md`

---

### 概念 2：里程碑交付制

**問題**：
AI 做事時，你不知道它在幹嘛。

**解決方案**：
拆分成「里程碑」，做完一個再報告。

#### 什麼是里程碑？

**不是**：每個小步驟  
**是**：一個可交付的成果

**範例**：

❌ **太碎**：
- 建立了檔案
- 寫了 10 行程式
- 測試了功能

✅ **里程碑**：
- JD 管理系統開發完成（能用了）
- BD 客戶開發表建立完成（有 16 家公司資料）

#### 交付流程

```
1. AI 做完一個里程碑
2. AI 測試確認（確保能用）
3. AI 用 3 行摘要報告你
4. 你說「OK」
5. AI 繼續下一個里程碑
```

---

### 概念 3：3 行原則

**問題**：
AI 回覆太囉唆，浪費時間。

**解決方案**：
預設 3 行，除非你說「解釋」。

#### 3 行包含什麼？

1. **完成了什麼**（結果）
2. **現在能幹什麼**（可用的東西）
3. **下一步是什麼**（計畫）

#### 範例對比

❌ **囉唆版**（200 字）：
```
我今天執行了 bd-automation.sh 腳本，
然後用 Python 爬取了 104 網站的招聘資訊，
接著使用 agent-browser 自動化了網頁操作，
提取了公司名稱、網址、電話、Email...
最後用 gog CLI 寫入了 Google Sheets...
```

✅ **簡潔版**（3 行）：
```
✅ 完成：BD 客戶開發表建立，16 家 AI 公司資料
📌 可用：[BD客戶開發表連結]
⏭️ 下一步：批量寄送 BD 信（6 家）
```

---

### 概念 4：授權機制

**問題**：
AI 每一步都問「可以嗎？」很煩。

**解決方案**：
建立「授權詞彙」。

#### 你的回覆 → AI 的理解

| 你說 | AI 理解 | AI 行動 |
|------|---------|---------|
| **OK** | 授權繼續 | 立即執行下一階段 |
| **解釋** | 要詳細說明 | 展開細節講清楚 |
| **不要** | 方向錯了 | 停止並重新思考 |
| **（沉默）** | 忙或在思考 | 等待，不催促 |

**關鍵**：AI 不應該一直問，做完再報告。

---

### 概念 5：每日日報

**問題**：
你得每天主動問「今天做了什麼？」

**解決方案**：
AI 每天 21:00 自動報告。

#### 設定日報 Cron Job

```javascript
{
  "name": "每日日報",
  "schedule": {
    "kind": "cron",
    "expr": "0 21 * * *",
    "tz": "Asia/Taipei"
  },
  "payload": {
    "kind": "systemEvent",
    "message": "整理今日完成進度，發送日報"
  }
}
```

#### 日報格式

```
📊 日報 (2026-02-10)

✅ 完成
- [結果描述]

📌 可用/交付
- [連結/檔案/系統]

⏳ 進行中
- [任務]：預計完成時間

🔴 卡點
- [問題]：需要什麼

⏭️ 明天計畫
- [任務]
```

## 5. 高效合作模式

### 模式 1：定方向，讓 AI 想辦法

**你做的事**：
- 定目標（「這週讓獵頭流程跑起來」）
- 做決策（選方案 A 或 B）
- 審核成果（里程碑交付時檢查）

**AI 做的事**：
- 拆分任務
- 想解決方案
- 執行並交付

**溝通方式**：
```
你: 這週讓獵頭流程跑起來
AI: 收到，拆成 4 天計畫：
    Day 1: 寄 BD 信（6家）
    Day 2: 測試履歷處理
    Day 3: 整合人選搜尋
    Day 4: 完整流程測試
    開始嗎？
你: OK
AI: [Day 1 開始執行]
```

---

### 模式 2：遇到問題，AI 先嘗試

**不好的方式**：
```
AI: 遇到問題，怎麼辦？
你: [需要花時間思考]
```

**好的方式**：
```
AI: 遇到問題，我試了 3 個方案：
    1. 修爬蟲（1小時，可能不穩定）
    2. 手動補充（穩定但慢）
    3. 先用現有資料（最快）
    建議選 3，你覺得？
你: 3
```

**原則**：AI 提供選項，你做選擇。

---

### 模式 3：大任務前先確認

**為什麼？**
- 避免做錯方向
- 管理期望值（預估時間）

**範例**：
```
AI: 準備建立獵頭完整系統
    預估需要：
    - 8 個工具腳本
    - 完整文件
    - GitHub Repo
    - 2-3 小時
    開始嗎？
你: OK
```

---

### 模式 4：透明記錄，隨時可查

**所有重要資訊都在檔案裡**

```
~/my-ai-assistant/
├── MEMORY.md                    # 長期記憶
├── memory/
│   ├── 2026-02-10.md            # 今天的日誌
│   └── ...
├── tasks/
│   ├── 待辦清單.md
│   └── 交接_狀態總覽.md
└── ...
```

**好處**：
- 你隨時可以查看進度
- AI 重啟後能快速恢復上下文
- 團隊成員能快速了解狀況

## 6. 實戰案例

### 案例 1：獵頭自動化系統

**目標**：
減少 70% 的 BD 客戶開發時間

**實作**：

**成果**：
- BD 開發時間：150 分鐘 → 30 分鐘（省 80%）

**關鍵**：
- 里程碑交付（每天交付一個可用的工具）
- 實際測試（不理想化，用真實資料測試）
- 誠實評估（做得到說做得到，做不到說做不到）

### 案例 2：專案管理

**目標**：
讓 AI 主動追蹤專案進度

**實作**：

#### 設定定時任務
```javascript
// 每天 9:00 檢查待辦
{
  "name": "每日待辦檢查",
  "schedule": {"kind": "cron", "expr": "0 9 * * *"},
  "payload": {
    "kind": "systemEvent",
    "message": "檢查今日待辦事項"
  }
}

// 每天 21:00 日報
{
  "name": "每日日報",

  }
}
```

**成果**：
- 不需要每天問「今天做了什麼？」
- AI 主動提醒該做的事
- 專案進度透明可查

## 7. 常見問題

### Q1: AI 還是很蠢，只會等指令怎麼辦？

**A**: 檢查這 3 件事：

1. **有沒有設定 SOUL.md？**
   - 如果沒有，AI 不知道自己該主動

2. **有沒有設定每日日報？**
   - 如果沒有，AI 不會主動報告

3. **你有沒有用「里程碑交付制」？**
   - 如果你每一步都問，AI 就會每一步都等你

**解決方式**：
按照本說明書的「快速開始」重新設定。

---

### Q2: AI 回覆太囉唆，怎麼辦？

**A**: 在 SOUL.md 加上這段：

```markdown
## 回報原則

1. 預設 3 行（完成了什麼、現在能幹什麼、下一步）
2. 不寫過程細節（不說「執行了XX指令」）
3. 只寫結果（能幹什麼、產出什麼）
4. 除非用戶說「解釋」，否則不展開細節
```

然後對 AI 說：「以後回報按照 SOUL.md 的規則，3 行原則」

---

### Q3: AI 每次重啟都忘記，怎麼辦？

**A**: 確保所有重要資訊都在檔案裡。

**檢查清單**：
- [ ] 有 MEMORY.md 嗎？
- [ ] 有 memory/YYYY-MM-DD.md 嗎？
- [ ] AI 每次啟動有讀取這些檔案嗎？

**測試方式**：
```
你: 你記得昨天我們做了什麼嗎？
AI: [應該能說出昨天的事]
```

如果 AI 說不出來，表示它沒讀取 memory 檔案。

---

### Q4: 如何讓 AI 主動做事（不等指令）？

**A**: 設定定時任務（Cron Job）

**範例 1：每天檢查待辦**
```javascript
{
  "name": "每日待辦檢查",
  "schedule": {"kind": "cron", "expr": "0 9 * * *"},
  "payload": {
    "kind": "systemEvent",
    "message": "檢查今日待辦事項並提醒"
  }
}
```

**範例 2：每小時檢查信箱**
```javascript
{
  "name": "信箱檢查",
  "schedule": {"kind": "every", "everyMs": 3600000},
  "payload": {
    "kind": "agentTurn",
    "message": "檢查信箱是否有重要郵件"
  }
}
```

---

### Q5: AI 做事方向錯了，怎麼辦？

**A**: 立即說「停」，然後重新定方向。

**範例**：
```
AI: [正在做 A、B、C...]
你: 停！我只要 A，其他不需要
AI: 收到，停止 B 和 C，只做 A
```

**教訓記錄**：
把這次經驗記錄在 MEMORY.md：
```markdown
## 教訓
- AI 容易做太多，要明確說「只要 A」
```

下次 AI 就會記得。

## 8. 進階技巧

### 技巧 1：建立「技能庫」

**為什麼？**
AI 不會每次都重新學習，讓它有「技能包」可以重複使用。

**範例**：
```
skills/
├── headhunter/           # 獵頭技能
│   ├── SKILL.md
│   └── scripts/
├── marketing/            # 行銷技能
│   ├── SKILL.md
│   └── templates/
└── ...
```

**使用方式**：
```
你: 執行獵頭 BD 開發流程
AI: [讀取 skills/headhunter/SKILL.md]
    [自動執行腳本]
    [完成並報告]
```

---

### 技巧 2：多 AI 分工

**為什麼？**
不同專長的任務，用不同的 AI。

**範例**：
- **YuQi (YQ1)** — 程式開發、專案管理
- **HR YuQi (YQ2)** — 招聘、HR 相關
- **行銷 YuQi (YQ3)** — 短影音、社群行銷

**共享**：
- MEMORY.md（共同記憶）
- skills/（技能庫）

**獨立**：
- SOUL.md（各自人設）

---

### 技巧 3：建立「教訓記錄」

**為什麼？**
AI 會犯錯，但不要讓它犯第二次。

**範例**：
```markdown
# MEMORY.md

## 教訓記錄

### 2026-02-10: 過度樂觀
- 問題：AI 說「明天就能完全自動化」
- 結果：做不到，失去信任
- 教訓：要誠實評估，說「需要 1-2 週測試調整」

### 2026-02-09: 背景做太多
- 問題：AI 背景做了 A、B、C、D...
- 結果：只需要 A，其他浪費時間
- 教訓：里程碑交付，做完一個再做下一個
```

AI 每次啟動會讀取 MEMORY.md，就不會重複犯錯。

---

### 技巧 4：用「實際可行性評估」

**為什麼？**
AI 容易理想化，要強迫它實際評估。

**範例**：
```
你: 評估這個系統實際能省多少時間
AI: [理想版] 能省 90% 時間！
你: 不要理想化，實際評估
AI: [實際版]
    - 短期：省 30-40%
    - 中期：省 50-60%（需調整）
    - 長期：省 70-80%（需 3 個月）
```
