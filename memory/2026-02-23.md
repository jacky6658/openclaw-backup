#### 3. 畫像匹配分析器 ✅
**檔案**：`match-personas.py`（15.2KB，600+ 行）

**匹配模型**：
```python
適配度 = (
  技能匹配 × 35% +
  成長匹配 × 25% +
  文化匹配 × 25% +
  動機匹配 × 15%
)

# 維度 1：技能匹配（35%）
技能匹配 = (
  技能組合匹配 × 50% +  # 候選人技能 vs 公司技術
  能力層級匹配 × 30% +  # 技術層級 vs 技術成熟度
  產業背景匹配 × 20%    # 產業經驗契合度
)

# 維度 2：成長匹配（25%）
成長匹配 = (
  職涯路徑匹配 × 50% +  # 動機 vs 公司成長路徑
  學習機會匹配 × 30% +  # 技術成熟度 vs 學習需求
  晉升速度匹配 × 20%    # 工作風格 vs 晉升速度
)

# 維度 3：文化匹配（25%）
文化匹配 = (
  工作風格匹配 × 40% +  # 技術宅 vs 自主型（100分）
  公司階段匹配 × 30% +  # 穩定型 vs 穩定企業（100分）
  用人風格匹配 × 30%    # 同工作風格匹配
)

# 維度 4：動機匹配（15%）
動機匹配 = (
  主要動機滿足度 × 60% +  # 想技術成長 vs 技術線（100分）
  不適配條件檢查 × 40%    # 不接受工地 vs 有工地（扣30分）
)
```

**評分標準**：
```
S 級（90-100分）: 完美匹配 - 強烈推薦
A 級（80-89分）: 高度匹配 - 推薦
B 級（70-79分）: 中度匹配 - 可考慮
C 級（60-69分）: 低度匹配 - 需評估
D 級（<60分）: 不匹配 - 不推薦
```

**輸出內容**：
- 總分 + 等級
- 4 維度詳細評分
- 適配亮點（3-5 條）
- 風險提示（含不適配條件）
- 建議：
  - 面試重點

  - 留任策略

### 與現有系統的關係

#### Talent Grading vs Persona Matching

**Talent Grading（人才評級）**：
```
輸入：候選人履歷
輸出：S/A+/A/B/C（候選人整體素質）
用途：評估候選人本身的質量（與職缺無關）

維度：
├── 技能匹配度 (25%)
├── 職涯發展軌跡 (25%)
├── 工作穩定性 (20%)
├── 工作年資 (15%)
├── 學歷背景 (10%)
└── 特殊加分 (5%)
```

**Persona Matching（畫像匹配）**：
```
輸入：候選人畫像 + 公司畫像
輸出：S/A/B/C/D（與特定公司/職缺的適配度）
用途：評估候選人與特定公司/職缺的契合度

維度：
├── 技能匹配 (35%)
├── 成長匹配 (25%)
├── 文化匹配 (25%)
└── 動機匹配 (15%)
```

**兩者結合使用**：
```
步驟 1：Talent Grading → 篩選 A 級以上候選人
步驟 2：Persona Matching → 與特定職缺匹配
步驟 3：推薦 → 同時考慮素質 + 適配度
```

#### AI Matcher v2 vs Persona Matching

**AI Matcher v2（舊系統）**：
```
只看硬技能：
├── 技能匹配 (40%)
├── 年資匹配 (30%)

問題：

優勢：

### 關鍵決策記錄

#### 1. 為什麼權重是 35/25/25/15？
```
技能匹配 (35%) - 最高權重
  └── 原因：技能不符直接淘汰，是基礎門檻

成長匹配 (25%) - 第二優先
  └── 原因：職涯路徑不符，候選人很快離職

文化匹配 (25%) - 同等重要
  └── 原因：文化不符，工作痛苦且難融入

動機匹配 (15%) - 最低權重
  └── 原因：動機可能隨時間改變，彈性最大
```

#### 2. 為什麼需要「不適配條件」？
```
硬性排除條件：
- 不接受工地 → 職缺需要工地 → 直接扣 30 分
- 不加班 → 公司高壓型 → 直接扣 30 分
- 不做協調 → 職缺需協調 → 直接扣 30 分

效益：
- 避免推薦不適合的候選人
- 降低面試浪費
- 提升 Offer 接受率
```

#### 3. 為什麼分「人才畫像」和「公司畫像」？
```
分離的好處：
1. 複用性：一個人才畫像可配對多個公司
2. 複用性：一個公司畫像可配對多個候選人
3. 可追蹤：候選人畫像隨時間更新（面試後補充）
4. 可追蹤：公司畫像隨企業發展更新
5. 批量配對：batch-match 直接使用
```

### 匿名化原則

#### 1. 人選姓名匿名化
```
選項 A（推薦）：英文代號
- Abeni, Sarah, Kevin, David...

- PM-2026-001, BE-2026-002...

```

#### 2. 公司名稱匿名化
```
原則：
✅ 保留產業類型（遊戲、電商、金融）
✅ 保留公司規模（30-100人、500-1000人）
✅ 保留業務性質（網路服務、電子製造）
❌ 移除品牌名稱

範例：
遊戲橘子 → 某遊戲科技公司（網路服務業，30-100人）
PChome → 某知名電商平台（電子商務，500-1000人）

為什麼保留產業和規模？
- 客戶需要評估候選人背景（大公司 vs 新創）
- 產業經驗很重要（金融業 vs 遊戲業）
- 不會洩露候選人身份（台灣有數百家遊戲公司）
```

- 學歷是公開資訊

- 不會洩露候選人身份（同校同系每年上百人）

### 關鍵決策記錄

#### 為什麼使用 Python 而非 Node.js？
```
優勢：
✅ 字串處理更強（匿名化規則）
✅ JSON 處理更簡潔
✅ 易於擴充規則
✅ 可獨立測試（不依賴後端）

劣勢：
❌ 需要 subprocess 呼叫
❌ 多一層語言

結論：優勢大於劣勢，採用 Python
```

#### 為什麼保留公司規模和產業？
```
客戶需要評估：
- 大公司 vs 新創？
- 金融業 vs 遊戲業？
- 高度專業 vs 通用技能？

同時不洩露身份：
- 某遊戲科技公司（30-100人）← 台灣有數百家
- 某金融機構（1000+人）← 無法精確判斷是哪一家
```

#### 為什麼範本要統一？
```
品牌一致性：
✅ 客戶體驗一致
✅ 專業形象統一
✅ 降低維護成本
✅ 所有 Bot 生成格式相同

禁止：
❌ 自行修改範本格式
❌ 使用其他範本
❌ 遺漏保密聲明
