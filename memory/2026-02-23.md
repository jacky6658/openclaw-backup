#### 2. Edge Cases 邊緣情況處理 (`edge-cases.md`, 6.2KB)

**7 種特殊情況**：

1. **社會新鮮人**（0 年經驗）
   - 學歷背景權重 ↑ 40%
   - 年資給予「潛力分」5-8 分
   - 實習經驗視為 0.5 年
   - 技能看學習能力（side projects, certifications）

2. **只有一份工作**
   - 穩定性基準分：60 分
   - <1 年：-10 分
   - 1-2 年：+0 分
   - 3+ 年：+10 分

3. **頻繁跳槽**（>5 次工作，<5 年）
   - 穩定性基準分：40 分
   - 每次 <1 年任期：-5 分
   - 檢查是否為「合理跳槽」（薪資成長 >30%、職位晉升）

4. **技能缺失**（skills = []）
   - 從職位名稱推斷（"AI 工程師" → ["Python", "ML"]）
   - 從工作經歷推斷（公司 = "Google" → +大廠經驗）
   - 最低給予 8 分（避免 0 分）

5. **工作經歷格式錯誤**
   - JSON parse 失敗 → 使用空陣列 []
   - 缺少 startDate/endDate → 估算為 1 年
   - 缺少 title → 視為「一般職員」

6. **國外學歷**
   - QS Top 50 → 視為名校（+3 分）
   - 未知學校 → 依學位等級（碩士 16 分）
   - 線上學位 → -2 分

7. **長期待業**（最後離職 >1 年）
   - 穩定性 -10 分
   - 檢查是否為「合理原因」（進修、創業、家庭）
   - 若有說明 → 不扣分

## [YQ1] 面試後候選人資料自動更新討論 ✅ (20:45-21:00)

### 背景
Phase 1 完成後，討論如何自動化面試後的候選人資料更新流程

### 現有流程問題
**手動更新流程**：
1. 獵頭顧問完成面試
2. 整理面試筆記
3. 手動登入系統
4. 逐一填寫欄位
5. 提交更新

**問題**：
- 耗時（每位候選人 5-10 分鐘）
- 容易遺漏細節
- 格式不一致
- 無法即時更新

### 提議方案：Bot 輔助自動更新

**新流程**：
```
獵頭顧問完成面試
  ↓
Telegram 傳送面試筆記給 Bot
  ↓
Bot 自動整理（AI 解析）
  ↓
結構化資料（JSON）
  ↓
自動呼叫 API 更新系統
  ↓
（可選）觸發重新評級
  ↓
Telegram 通知更新完成
```

### 技術架構設計

**1. 輸入格式（Telegram 訊息）**：
```
@HRyuqi_bot 更新候選人 #228
面試日期：2026-02-23
職位：AI工程師
面試官：Jacky
表現評價：優秀
技術能力：Python 熟練，有 ML 專案經驗
溝通能力：流暢，表達清晰
期望薪資：100k-120k
可到職日：一個月後
備註：對職位很有興趣，團隊 fit 良好
```

**2. Bot 處理邏輯**：
- 解析 Telegram 訊息（提取關鍵資訊）
- 結構化為 JSON
- 驗證資料完整性
- 呼叫 PUT /api/candidates/:id
- 可選：觸發 POST /api/candidates/:id/regrade

**3. API 端點（已存在）**：
- `PUT /api/candidates/:id` - 更新候選人資料
- `POST /api/candidates/:id/regrade` - 重新評級

### 資料欄位對應

**面試筆記 → 系統欄位**：
| 面試筆記 | 系統欄位 | 範例 |
|----------|---------|------|
| 面試日期 | lastContactDate | 2026-02-23 |
| 表現評價 | notes (追加) | "面試表現：優秀" |
| 技術能力 | notes (追加) | "技術：Python 熟練，ML 經驗" |
| 溝通能力 | notes (追加) | "溝通：流暢清晰" |
| 期望薪資 | notes (追加) | "期望：100k-120k" |
| 可到職日 | notes (追加) | "到職：一個月後" |
| 狀態 | status | 待聯繫 → 已聯繫 → 面試中 |
